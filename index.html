1. Describe the dataset and any issues with it¶

In this dataset, each row denotes a transaction and each column denotes an attribute corresponding to them. 
In detail, columns are: 
    step: a variable related to time for each transaction. 
    type: type of transaction, including "PAYMENT", "TRANSFER", "CASH_OUT", "CASH_IN" and "DEBIT". 
    amount: size of transactions. 
    nameOrig: holder of originating account. I noticed that these "names" always start by a character "C" or "M", possibly denotes clients and merchants. 
    oldbalanceOrg: balance of originating account before making the transaction. 
    newbalanceOrig: balance of originating account after making the transaction. 
    nameDest: holder of destination account. 
    oldbalanceDest: balance of destination account before making the transaction. 
    newbalanceDest: balance of destination account after making the transaction. 
    isFraud: identifier showing whether the transaction is a fraud. 
    isFlaggeredFraud: identifier showing whether the transaction is a potential fraud predicted by a model.

Before actually diving into the dataset, I did find an obvious issue with it, that after doing transactions like "TRANSFER" or "PAYMENT", the balance of destination account does not increase, specifically related to those merchant accounts. One potential reason of this can be the lack of data of these accounts. Althoughthere is no NULL values in this dataset, the 0 values may cause problems because we do not know whether they are missing or indeed 0.

3. Feature set and Model training¶

Based on the observations we found above, it is time to build the feature set. We need to do two data cleaning here, the first one is about 'type'. Since 'type' includes several categories represented by strings, we need to encode it to variables that our classifier can read. Thus, I tried to create five additional boolean features based on the 'type' we have, which are 'isTRANSFER', 'isCASH_OUT', 'isCASH_IN', 'isDEBIT' and 'isPAYMENT'. For these attributes, 1 denotes true and 0 denotes false. In this case, we successfully reprsent the 'type' feature in a more meaningful way.

Another data cleaning is about one of our observation. Since account types matter when identifying frauds, we need to extract this feature from the account name because input the whole name would mislead the model as it contains too much redundant information. Therefore, I created a boolean feature called 'client', denoting whether the destination account for the transaction is a client account or not, trying to help identify frauds.

After cleaned the data, we can start training the model based on the feature we select. Here I only selected feature I found meaningful, which are 'client', 'type' and 'amount'. And the algorithms I used are Decisio Tree and Naive Bayes because they are efficient and robust.
